Redis 的核心命令执行是**单线程**的，但它并不是严格意义上的单线程(数据在内存中，读取瓶颈在于网络IO)

在Redis6.0引入了多线程处理网络读写，但是命令的执行仍然时单线程
客户端请求 ──▶ 多线程读取/解析 ──▶ 单线程命令执行 ──▶ 多线程写回响应

在管理系统内的数据方面相较于传统数据库
> [!NOTE]
传统数据库（MySQL）：
客户端发 SQL → 从磁盘读数据到内存 → CPU处理 → 结果写回磁盘
Redis：
客户端发命令 → 直接在内存里找到数据 → CPU处理 → 结果写回内存


如果在实例上设置里持久性，则在某个时间间隔有一个[fork进程](fork())（不会阻塞主线程），来生成数据持久化[RDB快照](RDB快照)或[AOF](AOF)
由于两者的优缺点所以Redis都用，在重启的时候优先AOF。
由于AOF文件会越来越大，Redis会采用[[AOF重写]]去优化（用fork进程防止堵塞）

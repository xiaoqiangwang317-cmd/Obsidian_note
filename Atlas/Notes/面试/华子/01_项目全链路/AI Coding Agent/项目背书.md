简历原话：
基于大模型的代码智能体系统(AICodingAgent) 2024– 至今 2020– 2024 独立开发 2026.03–2026.04 
项目简介：设计并实现一套以大模型为核心的代码智能体系统，使模型具备代码理解、 任务执行与工具调用能力。系统通过任务调度与执行隔离机制支持复杂开发任务的处 理，可自动化完成代码编写、文件操作、项目构建等开发任务。 技术栈：Python、LLM(Qwen)、自研Agent架构、ToolCalling、[[MCP协议]]、GitWorktree 
核心工作： 
• 设计并实现Agent执行闭环([[AgentLoop]])：实现推理-工具调用-结果校验的全流程 自动化，支持将复杂开发任务拆解为3-5步执行链，提升任务自动化能力。 
• 封装文件操作与命令执行等[[ToolCalling]]能力：实现Task调度与状态管理，支持 多步骤任务追踪、异常重试与执行日志记录。 
• 基于GitWorktree 实现多任务并行执行与执行环境隔离：支持2-4个任务并发运 行，有效避免分支冲突。 
• 基于MCP+ToolRouter构建插件化扩展体系：统一接入外部工具并配置权限校验， 提升系统扩展性与安全性。

**先说服自己再说服别人**

项目本质：**这个项目本质上是把大模型放进一个“可调度、可执行、可反馈”的开发任务系统里，让它不只是生成代码，而是真的能分步骤完成开发任务。**

Q1:做这个项目的意义：

我做这个项目的意义，不是为了重复造一个现成的商业 Agent，而是想从底层真正理解并实现代码智能体的核心机制。相比直接使用成熟产品，自己做一套系统会逼着我去解决任务拆解、工具调用、执行隔离、状态管理、失败重试这些关键问题。这个过程让我从“会使用 AI 编码工具”提升到“能设计和实现 AI Coding Agent 系统”。同时它也有现实价值，因为很多企业场景会要求私有化、权限可控和内部工具接入，而这些能力往往需要自研框架来支撑。

Q2:你的项目怎么实现的?
我主要按照以下三个方面去讲一下我的项目实现流程：
第一个是项目的整体架构：
	整体上我把系统拆成了5个模块，分别是第一任务入口层，第二Agent调度层，第三工具执行层，第四执行环境层，第五扩展接入层
然后我大概讲一下这五个模块都是做什么的，
任务接入层：主要用来接收用户输入的开发任务，比如说在代码开发过程中需要实现的具体开发工作，比如新增接口，修改需求定位到原有的代码流程里，就是怎么根据需求定位到代码模块里面的Controller-service-service实现类-mapper-到数据库整个流程的修改，同时他还需要记录仓库、分支、约束条件这些上下文
第二层就是Agent的调度层：他主要负责的就是驱动整个[[Agent Loop]],让模型按照“理解任务 -> 规划步骤 -> 调工具执行 -> 根据结果继续决策 ”的方式循环运行
第三层是工具执行层，他主要负责的就是把文件读写、代码搜索、命令执行、构建测试这些能力封装成标准Tool，供大模型通过Tool Calling调用
第四层是执行环境层，目的是基于Git Worktree给不同任务创建独立工作目录和分支，保证多任务并发时互不干扰
最后一层是扩展接入层，基于MCP+Tool Router做统一工具接入、权限校验和路由分发，方便后续扩展更多外部能力。
第二个大的模块就是他的核心流程是怎么跑起来的，整个系统的执行流程大概是这样的：
用户先提交任务->然后调度器去创建上下文->Agent拆解步骤->Tool Router分发工具 -> Executor执行 -> 结果回写上下文 -> Agent判断下一步 -> 直到完成
我举个例子来说的话就是：
用户先提交一个自然语言任务，比如“给任务加登录接口并补测试”，系统会先创建一个task_id,记录任务描述、目标仓库、可用工具、执行状态、日志信息。
接着调度器会先创建独立的 git worktree 和任务分支，把这次任务放到隔离环境里跑，避免直接污染主工作区
然后进入 Agent Loop
我让大模型先做任务拆解，而不是一上来就写代码，他会先规划出3-5个步骤，比如：
	扫描项目结构
	定位认证模块
	编写登录接口
	补测试
	运行测试并汇总结果
之后模型会按步骤发起Tool Calling
比如它要先搜索认证相关代码，就调用 search_code；  
要读取文件，就调用 read_file；  
要修改代码，就调用 write_file 或 patch 工具；  
要验证结果，就调用 run_command 去跑测试或构建。

每次工具执行完，我都会把结果回写到任务上下文里，包括：
- 工具输出
- 命令返回码
- 改动文件
- 错误日志
- 当前步骤状态
然后再把这些结果交回给模型，让它判断当前这一步是否完成、要不要补一步、要不要重试、要不要终止。  
这样就形成了一个完整闭环，而不是一次性生成。

第三个就是几个关键模块我是怎么设计的
**1. Agent Loop**
这是整个系统的核心
我没有把模型当成一次性代码生成器，而是把它放进一个循环里：

- Reason：根据任务和当前上下文判断下一步做什么
- Act：选择合适的工具并生成参数
- Observe：接收工具执行结果
- Reflect：判断成功、失败、重试还是继续

这个循环会一直执行，直到：

- 任务完成
- 达到最大步数
- 连续失败超过阈值

所以系统真正的关键不是模型本身，而是我怎么把模型放进一个可控的执行循环里。

**2. Task Scheduler**  
调度器负责维护任务状态。  
我给任务定义了比较明确的状态，比如：

- pending
- running
- retrying
- failed
- completed

这样系统就能追踪每个任务当前在哪一步、失败了几次、是不是超时、是不是需要重试。  
如果并发执行多个任务，调度器还要负责队列管理和资源隔离。

**3. Tool Calling**  
我把底层能力都抽象成标准工具接口，让模型只能通过这些受控工具与外部环境交互。  
比如：

- read_file(path)
- write_file(path, content)
- search_code(keyword)
- run_command(command)
- create_worktree(branch)

这样设计有两个好处：
- 模型行为是结构化的，不是随便输出一段自然语言命令
- 工具层可以统一做参数校验、权限控制、日志记录

**4. Git Worktree 隔离执行**  
这是这个项目里比较有辨识度的一点。  
因为代码智能体经常要实际修改文件、运行命令，如果多个任务都在同一个工作区执行，很容易互相污染。

所以我用了 Git Worktree 给每个任务分配独立工作目录和分支。  
这样一个任务在改登录接口，另一个任务在修测试，它们可以并发进行，最后各自保留独立改动，不会互相覆盖。

这部分本质上解决的是：AI 执行任务时的工程隔离问题。

**5. MCP + Tool Router**  
为了让系统后续更容易扩展，我没有把所有工具都写死，而是设计了 MCP + Tool Router 的插件式接入方式。

Tool Router 负责：

- 注册工具
- 暴露统一描述给模型
- 校验参数
- 校验权限
- 把请求转发到具体工具实现

这样以后如果要接入更多工具，比如：

- lint
- test runner
- 构建系统
- 内部代码检索
- 文档查询

就不用重构 Agent 主流程，只要按协议接进来就行。

**四、它为什么能真正完成开发任务，而不是只会聊天**

因为我在实现上重点解决了 4 个问题：

1. 任务拆解  
    让模型先规划，再执行，避免大任务直接失控。
    
2. 工具约束  
    让模型通过结构化 Tool Calling 操作环境，而不是无约束输出命令。
    
3. 执行反馈  
    每一步都有真实结果返回，而不是模型自己假设“我已经改好了”。
    
4. 失败恢复  
    通过状态管理和重试机制，让系统能根据测试失败、命令报错继续修正，而不是一次失败就结束。
    

所以这个项目的本质不是“让大模型写代码”，而是：

**让大模型在一个受控的工程系统里，作为决策核心去驱动任务执行。**

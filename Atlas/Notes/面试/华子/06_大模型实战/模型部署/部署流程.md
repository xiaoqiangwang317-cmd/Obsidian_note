# 一、你最该懂的调参，不是训练超参，而是推理参数

你现在这个岗位方向，更常见的是：

**推理调参**，不是训练调参。

你先记住这几个最常见的：

---

## 1. temperature

控制回答的发散程度。

- 越低：越稳、越保守、越像标准答案
- 越高：越发散、越有创造性、越不稳定

你可以这样理解：

- 面试问答 / 知识检索：temperature 低一点
- 文案生成 / 创意写作：temperature 高一点

你面试时可以说：

**如果是企业知识问答或 Agent 工具调用，我会把 temperature 设低一点，优先保证稳定性和一致性。**

---

## 2. top_p

控制候选词采样范围。

它和 temperature 有点像，但更偏“截断候选空间”。

你现在不用讲太细，先记一句：

**一般不会同时把 temperature 和 top_p 都调得特别激进。**

---

## 3. max_tokens / 输出长度

控制模型最多生成多少内容。

意义很现实：

- 太短：回答不完整
- 太长：浪费时间和成本，还容易跑偏

你面试可以说：

**在 Agent 或问答场景里，输出长度需要限制，否则会影响响应时延和成本。**

---

## 4. stream

是否流式返回。

- stream=false：等完整结果一起返回
- stream=true：边生成边返回

你现在的项目里已经有这个概念了，所以很好讲：

**如果用户很关注交互体验，我会用流式输出；如果更关注结构化结果，就可以一次性返回。**

# 二、部署时真正重要的不是“神秘调参”，而是这些工程取舍

这个比参数更重要。

---

## 1. 模型大小怎么选

你不能一上来就上超大模型。

你要会讲：

- 小模型：便宜、快、容易本地部署
- 大模型：能力强，但资源消耗高

你这次部署 Qwen2.5:3b 就是一个很好的例子。

你可以说：

**我这次先选 3B 级别模型，是为了快速验证本地部署和推理闭环；如果业务效果不够，再考虑更大参数模型。**

---

## 2. 本地部署 vs 云 API

这是高频问题。

### 本地部署优点

- 数据更可控
- 成本可预测
- 更适合私有化场景
- 可以做更细的系统级集成

### 本地部署缺点

- 部署维护更复杂
- 算力有限
- 模型能力可能不如顶级闭源模型

### 云 API 优点

- 上手快
- 能力强
- 不用自己维护模型服务

### 云 API 缺点

- 成本可能更高
- 数据合规敏感
- 可控性较弱

这个你一定要会讲。

---

## 3. 响应速度和资源占用

部署时面试官也可能会问：

- 为什么本地模型慢？
- 为什么显存/内存占用大？
- 为什么要量化？

你现在先记一句够用了：

**模型越大，推理延迟和资源消耗通常越高，所以部署时常常会通过选小模型、量化和限制输出长度来平衡效果与成本。**

[[语言表达]]

**项目经历**

1. [[“中科院软件所 - GraphRAG 项目”]]

这是你面对算法/大模型团队最有技术深度的项目，必须重点拿下。

- **Input (痛点)**：当时我们在做垂直领域的问答系统，最大的痛点是单纯依靠向量检索（Vector RAG），大模型的幻觉非常严重，并且处理长上下文时，系统的 HTTP 响应经常超时阻塞 。
    
- **Processing (架构方案)**：为了解决这个问题，我参与设计了一套混合检索链路和高吞吐的异步架构。底层存储我用了 Neo4j 图数据库加上 Elasticsearch 倒排索引 。
    
- **Deep Dive (核心技术 - 面试官最爱听的)**：
    
    1. **混合检索逻辑**：我设计了“图谱拓扑关联+全文语义检索”的双重召回机制。先把实体间的关系用图谱拉出来，再结合 ES 的语义匹配，相当于给了大模型一个非常确定的知识边界 。
        
    2. **异步流式改造**：面对 I/O 阻塞，我没有用传统的同步等待，而是引入 RocketMQ 做消息解耦，配合 WebSocket 把大模型的 Token 进行流式（Streaming）推送到前端 。
        
- **Output (结果)**：这套架构不仅显著降低了模型在垂直领域的幻觉，还彻底解决了长任务堆积导致的系统瓶颈，极大优化了用户的首字延迟体验 。
    

2. [[讲透“智能客服多 Agent 系统”]]

这个项目体现了你跳出框架限制，做底层设计的工程素养。

- **Input (痛点)**：复杂的客服场景如果用单一大模型，意图识别和任务执行很容易混乱。而当时主流的 LangChain 等框架太黑盒，排查问题极度困难 。
    
- **Processing (架构方案)**：所以我决定摒弃黑盒，直接用原生 Python 写底层逻辑。我设计了一个多 Agent 协同工作流，把意图理解、知识检索、答案生成拆分成独立的节点 。
    
- **Deep Dive (核心技术)**：
    
    1. **状态机调度**：我是基于手写状态机来实现调度的。每一次对话流转，都在状态机中有明确的记录，这使得复杂任务的调度实现了精细化管理，扩展性极强 。
        
    2. **工程化 RAG**：在知识检索节点，我搭建了 Embedding + FAISS 的检索系统，并且针对不同的 Agent 编写了差异化的 Prompt，用来收敛模型的发散思维 。
        
- **Output (结果)**：最终通过 FastAPI 标准化输出，底层用 MySQL 和 Redis 做上下文记忆持久化，跑通了一个高可用、低幻觉的多轮对话服务闭环 。
